Inducción Matemática
📖 Teoría
🎮 Visualización
✏️ Quiz
🎯 Problema Motivador

Imagina que quieres demostrar que la suma 1 + 2 + 3 + … + n = n(n+1)/2 es verdadera para todos los números naturales. Verificarla para n=1, n=2, n=100 no es suficiente — necesitamos una herramienta que cubra infinitos casos de una sola vez. Esa herramienta es la inducción matemática.

💡 Intuición — El efecto dominó

Piensa en una fila infinita de fichas de dominó. Si sabes dos cosas:

(1) La primera ficha cae, y (2) cada vez que una ficha cae, la siguiente también cae...

...entonces todas las fichas caerán. La inducción matemática funciona exactamente igual.

Estructura de una Prueba por Inducción
BASE Caso Base — n = n₀
Demostrar que la proposición P(n₀) es verdadera para el valor inicial (usualmente n=0 o n=1).
Es el equivalente a empujar la primera ficha de dominó.
HIP Hipótesis Inductiva
Asumir que P(k) es verdadera para algún k ≥ n₀ arbitrario pero fijo.
No estamos probando P(k) — la estamos asumiendo para usarla después.
PASO Paso Inductivo — P(k) → P(k+1)
Usando la hipótesis inductiva, demostrar que P(k+1) también es verdadera.
Este es el corazón de la prueba: si una ficha cae, la siguiente también cae.
Ejemplo 1 — Suma de los primeros n enteros
Proposición: Para todo n ≥ 1: 1 + 2 + 3 + … + n = n(n+1)/2
BASE n = 1
Lado izquierdo: 1
Lado derecho: 1·(1+1)/2 = 1·2/2 = 1 ✓
El caso base es verdadero.
HIP Hipótesis para n = k
Asumimos que: 1 + 2 + … + k = k(k+1)/2
PASO Demostrar para n = k+1
1 + 2 + … + k + (k+1)
= [1 + 2 + … + k] + (k+1)    ← separamos el último término
= k(k+1)/2 + (k+1)          ← aplicamos la hipótesis inductiva
= (k+1)[k/2 + 1]
= (k+1)(k+2)/2              ← que es exactamente la fórmula para n=k+1 ✓
Ejemplo 2 — Potencias de 2
Proposición: Para todo n ≥ 0: 2⁰ + 2¹ + … + 2ⁿ = 2ⁿ⁺¹ − 1
BASE n = 0
Lado izquierdo: 2⁰ = 1
Lado derecho: 2⁰⁺¹ − 1 = 2 − 1 = 1 ✓
HIP Hipótesis para n = k
Asumimos: 2⁰ + 2¹ + … + 2ᵏ = 2ᵏ⁺¹ − 1
PASO Para n = k+1
2⁰ + 2¹ + … + 2ᵏ + 2ᵏ⁺¹
= (2ᵏ⁺¹ − 1) + 2ᵏ⁺¹    ← hipótesis inductiva
= 2·2ᵏ⁺¹ − 1
= 2ᵏ⁺² − 1 = 2⁽ᵏ⁺¹⁾⁺¹ − 1 ✓
Ejemplo 3 — Desigualdad n! > 2ⁿ
Proposición: Para todo n ≥ 4: n! > 2ⁿ
BASE n = 4
4! = 24  >  2⁴ = 16 ✓
HIP Hipótesis para n = k ≥ 4
Asumimos: k! > 2ᵏ
PASO Para n = k+1
(k+1)! = (k+1) · k!
> (k+1) · 2ᵏ    ← hipótesis inductiva
> 2 · 2ᵏ       ← porque k+1 > 2 para k ≥ 4
= 2ᵏ⁺¹ ✓
Ejemplo 4 — Inducción Fuerte: Fibonacci
Proposición: El n-ésimo número de Fibonacci satisface F(n) < 2ⁿ para todo n ≥ 1
🔷 ¿Qué es la Inducción Fuerte?

En la inducción fuerte (o completa), la hipótesis inductiva asume que P(j) es verdadera para todos los valores j con 1 ≤ j ≤ k, no solo para j = k. Esto es útil cuando el paso inductivo necesita más de un caso anterior.

Recordatorio: F(1) = 1, F(2) = 1, F(n) = F(n-1) + F(n-2) para n ≥ 3.

BASE n = 1 y n = 2
F(1) = 1 < 2¹ = 2 ✓
F(2) = 1 < 2² = 4 ✓
Necesitamos dos casos base porque el paso inductivo usará F(k-1) y F(k-2).
HIP Hipótesis Fuerte para todo j ≤ k
Asumimos que F(j) < 2ʲ para todo j con 1 ≤ j ≤ k (donde k ≥ 2).
En particular: F(k) < 2ᵏ y F(k-1) < 2ᵏ⁻¹
PASO Demostrar para n = k+1
F(k+1) = F(k) + F(k-1)    ← definición de Fibonacci
< 2ᵏ + 2ᵏ⁻¹                ← hipótesis fuerte aplicada a k y k-1
< 2ᵏ + 2ᵏ                  ← porque 2ᵏ⁻¹ < 2ᵏ
= 2·2ᵏ = 2ᵏ⁺¹ ✓
⚠️ Error común: Olvidar verificar el caso base. Si el caso base falla, toda la prueba es inválida aunque el paso inductivo sea correcto. También es un error asumir P(k+1) en lugar de P(k) en la hipótesis inductiva.
🔗 Conexión: La inducción matemática es la base para probar correctitud de algoritmos recursivos, que veremos en el Bloque 4 al analizar recurrencias.

✏️ Quiz
🎯 Problema Motivador

Tienes dos algoritmos para ordenar una lista. Uno tarda 3n² + 100n + 500 operaciones y otro tarda 0.001n³. ¿Cuál es más eficiente? Para n=10, el primero tarda 1500 y el segundo 1. Pero para n=1000, el primero tarda ~3 millones y el segundo ~1000 millones. La notación asintótica nos permite comparar algoritmos ignorando constantes y enfocándonos en el comportamiento a largo plazo.

💡 Intuición — Antes de la definición formal

Imagina que estás comparando velocidades de dos corredores. No te importa si uno lleva zapatillas Nike y el otro Adidas (las constantes). Lo que importa es: ¿quién llega primero cuando la distancia es muy grande?

Las 5 Notaciones
NotaciónNombreIntuiciónLímite f(n)/g(n)
O(g) Big-O f crece a lo mucho tan rápido como g lim = 0 ó c > 0
Ω(g) Big-Omega f crece al menos tan rápido como g lim = c > 0 ó ∞
Θ(g) Big-Theta f crece exactamente tan rápido como g lim = c > 0
o(g) Small-o f crece estrictamente más lento que g lim = 0
ω(g) Small-omega f crece estrictamente más rápido que g lim = ∞
Definiciones Formales
Big-O: f(n) ∈ O(g(n)) ⟺ ∃ c > 0, n₀ > 0 : ∀ n ≥ n₀, f(n) ≤ c·g(n)
Big-Ω: f(n) ∈ Ω(g(n)) ⟺ ∃ c > 0, n₀ > 0 : ∀ n ≥ n₀, f(n) ≥ c·g(n)
Big-Θ: f(n) ∈ Θ(g(n)) ⟺ f(n) ∈ O(g(n)) ∧ f(n) ∈ Ω(g(n))
Small-o: f(n) ∈ o(g(n)) ⟺ ∀ c > 0, ∃ n₀ : ∀ n ≥ n₀, f(n) < c·g(n)   ⟺   lim[n→∞] f(n)/g(n) = 0
Small-ω: f(n) ∈ ω(g(n)) ⟺ ∀ c > 0, ∃ n₀ : ∀ n ≥ n₀, f(n) > c·g(n)   ⟺   lim[n→∞] f(n)/g(n) = ∞
Jerarquía de Funciones
Propiedades Importantes
📏 Regla de la Suma

O(f(n) + g(n)) = O(max(f(n), g(n)))

Ejemplo: O(n² + n) = O(n²). El término de mayor orden domina.

✖️ Regla del Producto

O(f(n)) · O(g(n)) = O(f(n) · g(n))

Ejemplo: O(n) · O(log n) = O(n log n).

🔗 Transitividad

Si f(n) = O(g(n)) y g(n) = O(h(n)), entonces f(n) = O(h(n)).

⚠️ Error común: Confundir O con Θ. Decir "el algoritmo es O(n²)" no implica que sea Θ(n²). Un algoritmo O(n²) podría ser Θ(n) en el mejor caso. Siempre especifica si hablas del peor, mejor o caso promedio.

🎯 ¿Qué es un Grafo?

Un grafo G = (V, E) es una estructura matemática compuesta por un conjunto de vértices V (también llamados nodos) y un conjunto de aristas E (también llamadas arcos o edges) que conectan pares de vértices.

Los grafos modelan relaciones entre objetos: redes sociales, mapas de ciudades, circuitos eléctricos, dependencias entre tareas, y muchos más.

Tipos de Grafos
TipoDescripciónEjemplo

Dirigido (dígrafo)Las aristas tienen dirección (u → v)Páginas web con hipervínculos
PonderadoCada arista tiene un peso o costoMapa de carreteras con distancias

Terminología Esencial
📚 Conceptos clave

Grado de un vértice: número de aristas incidentes en él. En grafos dirigidos: grado de entrada (in-degree) y grado de salida (out-degree).

Camino: secuencia de vértices v₁, v₂, …, vₖ donde cada par consecutivo está conectado por una arista.

Ciclo: camino donde el vértice inicial y final son el mismo.

Componente conexa: subconjunto maximal de vértices donde existe camino entre todo par.

Las 4 Representaciones
1️⃣ Matriz de Adyacencia

Una matriz A de tamaño |V| × |V| donde A[i][j] = 1 si existe arista de i a j, y 0 en caso contrario.

Espacio: O(V²) — siempre ocupa V² celdas independientemente del número de aristas.

Cuándo usar: grafos densos (muchas aristas), cuando necesitas verificar si existe una arista en O(1).

Propiedad: En grafos no dirigidos, la matriz es simétrica (A[i][j] = A[j][i]).

2️⃣ Lista de Adyacencia

Un arreglo de |V| listas, donde la lista del vértice i contiene todos los vértices adyacentes a i.

Espacio: O(V + E) — proporcional al tamaño real del grafo.

Cuándo usar: grafos dispersos (pocas aristas), cuando necesitas iterar sobre vecinos eficientemente.

Es la representación más usada en algoritmos de grafos (BFS, DFS, Dijkstra, etc.).

3️⃣ Matriz de Incidencia

Una matriz de |V| × |E| donde M[i][j] = 1 si el vértice i es extremo de la arista j.

Espacio: O(V · E) — generalmente más costosa.

Cuándo usar: útil para multigrafos (múltiples aristas entre el mismo par de vértices).

4️⃣ Lista de Aristas (Edge List)

Una lista de pares (u, v) — o tripletas (u, v, w) si es ponderado — representando cada arista.

Espacio: O(E).

Cuándo usar: cuando el algoritmo itera sobre todas las aristas (ej. algoritmo de Kruskal para árbol de expansión mínima).

🎯 ¿Cómo analizamos un algoritmo?

El análisis de algoritmos busca responder: ¿cuántos recursos (tiempo, espacio) consume un algoritmo en función del tamaño de la entrada n?

Usamos el modelo de costo: cada operación elemental (asignación, comparación, operación aritmética) cuesta O(1). Contamos cuántas operaciones elementales ejecuta el algoritmo en el peor caso.

🎯 ¿Cómo pasar de un problema a un algoritmo?

El diseño de algoritmos es un proceso sistemático: (1) entender el problema, (2) identificar la estrategia adecuada, (3) escribir el pseudocódigo, (4) verificar con ejemplos, (5) analizar la complejidad.

Estrategias de Diseño
EstrategiaCuándo usarEjemplo
Fuerza BrutaSolución simple, n pequeñoBuscar par que suma objetivo
BúsquedaEncontrar elemento con propiedadBúsqueda lineal, binaria
Dos PunterosArreglo ordenado, pares/subarreglosInvertir arreglo
Ventana DeslizanteSubarreglos contiguosSuma máxima de k elementos
ConteoFrecuencias, estadísticasContar pares, inversiones
Cómo Escribir Buen Pseudocódigo
✏️ Principios

1. Claridad sobre brevedad: El pseudocódigo debe ser legible por un humano, no ejecutable por una máquina.

2. Nivel de abstracción adecuado: No escribas código de máquina, pero tampoco seas tan abstracto que no se entienda el algoritmo.

3. Verificar con ejemplos: Traza el algoritmo manualmente con un ejemplo pequeño antes de afirmar que es correcto.

4. Nombrar variables descriptivamente: max_val en lugar de x, counter en lugar de c.
